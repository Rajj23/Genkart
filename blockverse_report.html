BlockVerse Project Report Abstract
Abstract

Modern organizational environments heavily demand efficient real-time synchronous document collaboration solutions. Popular consumer products such as Notion and Google Docs offer intuitive workflows but often operate as black-box SaaS applications with complex, proprietary infrastructure. This report details the research, architectural design, database design, and end-to-end cloud implementation of BlockVerse, an enterprise-ready, open-source real-time collaborative document editor.

BlockVerse utilizes a block-structured system architecture to break down rich-text documents into atomic granular items (paragraphs, headers, check-lists, code-blocks). To manage real-time text-concurrency sync and user presence updates, we developed a highly scalable message distribution layer. The backend relies on a **Java 21 / Spring Boot 3** ecosystem running Spring WebSocket protocols utilizing standard STOMP communication. Client-side state transitions are synchronized in sub-100ms latency windows.

To guarantee scalability under heavy synchronous volumes, WebSocket processes do not interact directly with persistent databases. Instead, messages are ingested into an **Apache Kafka** distributed messaging queue (running KRaft clustering mode without Zookeeper) and dispatched to active peer nodes. A **Redis 7.2 Cache** is deployed as an in-memory memory map to capture cursor shifts, typing flags, and cache active documents. Persistence is achieved through an **AWS RDS MySQL** instance, protected via JPA optimistic locking and transaction version tracking, with media uploads hosted via secure **AWS S3** object buckets.

The client is designed using **Next.js 16 (React 19)**, TypeScript, and **Tailwind CSS v4** to supply a premium, responsive, and responsive Notion-like workspace environment. DevOps operations are fully codified, using **Terraform Infrastructure as Code (IaC)** to lease AWS Virtual Private Clouds (VPC), EC2 instances, ECR repositories, and S3 resources. Deployment is fully containerized inside **Docker** multi-stage constructs and orchestrated via a lightweight **k3s Kubernetes** cluster configured with cert-manager Let's Encrypt TLS and automated Nginx Ingress routers. Automated deployments are fully handled using high-coverage **GitHub Actions** CI/CD pipelines.

Chapter 1: Introduction BlockVerse Report

1. Introduction

1.1 Project Overview & Paradigm Shift

Collaboration has transformed from a periodic business requirement into a continuous, real-time necessity. Synchronous editing architectures empower remote and distributed software engineering, product management, and corporate operations groups to synthesize knowledge concurrently without context-switching bottlenecks. Real-time document management tools must satisfy strict performance levels: cursor tracking must render immediately, character entry must match across workspaces instantly, and editing blocks must transition smoothly between devices without data loss or synchronization faults.

BlockVerse addresses this challenge. Developed as a self-contained, enterprise-grade Notion-style platform, BlockVerse redefines web-based collaborative document authoring through a modern block-level abstraction system. Rather than tracking documents as large, monolithic text files, BlockVerse structures each canvas as an ordered collection of granular, self-describing components called Blocks. Each Block possesses a unique identifier, distinct type characteristics (e.g., Code block, H2, Checklist item), positioning sequence, and optimistic database lock attributes.

1.2 Core Development Motivation

The motivation behind BlockVerse is to bridge the gap between heavy, enterprise document-sync platforms and developer-centric open architectures. Commercial collaborative editors rely on complex, closed SaaS models that restrict custom security configurations, direct database control, and flexible deployment models. Engineering groups need a self-hostable, performant alternative that integrates easily into containerized enterprise networks.

BlockVerse addresses these needs by combining high-performance Java frameworks with flexible TypeScript UI systems. Using Spring Boot's WebSocket engine alongside Apache Kafka message distribution and Redis caching, BlockVerse achieves sub-100ms sync speeds. This design keeps the persistence tier clean and responsive, providing a reliable self-hosted document hub.

1.3 Boundary, Scope, & Limitations

The current boundary of the BlockVerse platform includes:

  • Granular Multi-User Real-time Sync: Active cursor positions, user-typing indicators, block state transitions, and workspace activity feeds.
  • Comprehensive Workspace Management: Workspace scopes (personal vs. team), membership controls, permission audits, and document public sharing.
  • Fully Declarative Infrastructure: Automated Terraform templates, Multi-Stage Docker containers, and live k3s Kubernetes deployments on AWS.

Key Limitations: The current implementation relies on server-managed sequence resolution and optimistic locking via database versions, rather than peer-to-peer CRDT (Conflict-Free Replicated Data Types) engines, meaning concurrent modifications to a single block require rapid atomic sequence resolution at the backend layer.

2.2 Deep Investigation of Existing System Flaws

Existing commercial collaborative editing platforms present several operational challenges for modern enterprises:

  • Data Ownership and Sovereign Security: Enterprise data is stored in third-party clouds, creating compliance issues for medical, financial, and governmental organizations that require strict on-premises data isolation.
  • Monolithic Content Models: Systems like Google Docs store documents as large, monolithic objects. Modifying a single character requires serializing the entire document tree, which consumes excessive database bandwidth and complicates granular history auditing.

System administrators cannot inspect real-time user-level cursor latencies, WebSocket active connections, or database query execution plans. BlockVerse resolves this by integrating a native **Prometheus + Grafana** stack to track active WebSocket streams and host performance metrics.

Chapter 3: Problem Statement BlockVerse Report

3. Problem Statement & Analysis

3.1 Structured Problem Specification

Designing a real-time collaborative document editor requires solving three primary engineering challenges:

The Concurrency and Performance Challenge
When multiple authors edit different parts of a document at the same time, the server must process incoming changes, verify sequence numbers, update database records, and broadcast changes to other users without lag or display desynchronization.

Traditional architectures route collaborative transactions directly through transactional databases like PostgreSQL or MySQL. While this ensures ACID safety, it creates scalability limits. Under heavy loads, the database connection pool exhausts quickly, and row-level locks on tables can block processing threads, leading to application crashes and service interruptions.

BlockVerse resolves these scaling challenges by decoupling document edits from the core persistence layer. The system uses a multi-tiered architecture: real-time updates are queued in Apache Kafka, cached in a Redis in-memory layer, and persisted to MySQL asynchronously. This keeps database interactions fast, predictable, and resilient.

3.2 Feasibility Study

A multi-dimensional feasibility study was conducted to validate the project design:

  • Technical Feasibility: The combination of Java 21 Virtual Threads, Spring Boot WebSockets, Apache Kafka, and Next.js 16 represents a modern, proven architecture for handling thousands of concurrent connections.
  • Operational Feasibility: Operating the platform is simplified using containerized microservices. K3s Kubernetes automates container restarts, while Ingress manages SSL certificates and load balancing automatically.
  • Economic Feasibility: The entire stack runs efficiently on a single AWS EC2 instance (`c7i-flex.large`) combined with a `db.t3.micro` RDS database. This low-resource footprint provides enterprise-grade collaboration features at a fraction of the cost of commercial SaaS models.

5.2 Schema Design & ER Database Architecture

The database schema is structured to ensure fast loading times and clear isolation between user workspaces, documents, and individual content blocks.

Figure 5.2: Entity-Relationship Database Schema Layout

Optimistic concurrency control is managed at the document level. When a block is created, deleted, or reordered, the system validates the document version number. If the client-provided version matches the database version, the change is applied and the version is incremented. If the versions do not match, the system rejects the operation and updates the client with the latest server state to prevent overwriting concurrent edits.

5.3 Real-Time Sync Protocols & State Machines

Real-time updates are processed asynchronously through a WebSocket-based messaging pipeline to ensure low-latency communication.

Figure 5.3: Event-Driven Real-Time Synchronization Flow

Using Apache Kafka as a messaging backplane enables horizontal scaling of the real-time collaboration layer. Multiple instances of the Spring Boot backend can run simultaneously on different nodes in a Kubernetes cluster. When a user connects to a WebSocket port on a specific instance and broadcasts a change, that server publishes the update to the Kafka cluster. The other backend instances consume the message from the shared topic and broadcast it to their respective connected clients, ensuring consistent, low-latency document state synchronization across the entire network.

Chapter 6: Technology Stack BlockVerse Report

6. Technology Stack Details

The BlockVerse architecture relies on modern, robust technologies across every layer of the system.

6.1 Core Application Frameworks

Java 21 & Spring Boot 3

The backend services are developed using Java 21 and Spring Boot 3. The system takes advantage of Java 21's Virtual Threads (Project Loom) to process blocking database queries and I/O requests efficiently. By running blocking tasks on lightweight virtual threads rather than expensive system threads, the backend maintains steady performance under heavy workloads.

Next.js 16 (React 19) & Tailwind CSS v4

The frontend web application is built on Next.js 16, utilizing its App Router structure to separate public content from authenticated collaboration spaces. Next.js Server Components optimize page load times by rendering static elements on the server. The user interface styles are written in Tailwind CSS v4, which compiled styling rules to provide a highly responsive, performant user experience across all screen sizes.

6.2 Cache & Messaging Layers

Redis Cache Integration

A Redis 7.2 instance handles high-frequency data operations. User cursor positions, active editing states, and temporary operational locks are kept in Redis to avoid hitting the primary MySQL database. Using Redis for hot storage keeps the platform's core persistence layers fast and scalable.

Apache Kafka Event Backplane

Real-time collaborative events are synchronized via Apache Kafka. Operating in KRaft mode, Kafka manages system messaging without Zookeeper. Event logs are split into dedicated partitions, allowing backend server nodes to publish and subscribe to document change feeds concurrently, keeping user screens in sync in sub-100ms windows.

7. Implementation Architecture

BlockVerse employs robust architectural design patterns to ensure the clean separation of concerns and high system maintainability.

7.1 Layered Backend Design

The backend services are structured into distinct architectural layers: Controller, Service, Repository, DTO (Data Transfer Object), and Config components. Incoming REST requests and WebSocket connections are received by the Controller tier, validated, and passed down to the Service layer for business logic execution. Data access is handled by Spring Data JPA Repository interfaces, mapped directly to MySQL tables.

Package Name Target Responsibility Operational Details
com.blockverse.app.controller API endpoints & socket routing Exposes REST end-points and maps STOMP sockets
com.blockverse.app.service Transactional business logic Manages transaction scopes and coordinates Kafka producers
com.blockverse.app.repo Data persistence layer Abstracts database query logic utilizing JPA Hibernate
com.blockverse.app.entity Object Relational Mappings Maps database schemas to core Java classes

7.2 Decoupled Event-Driven Flow

When a user updates a block within a collaborative document, the transaction flows through the decoupled layers:

  1. The client app broadcasts a granular JSON edit message over a WebSocket connection to the `/app/document.update` endpoint.
  2. The `BlockSocketController` receives the STOMP message, parses the payload into a `BlockOperationDTO`, and forwards it to the `BlockService`.
  3. The `BlockService` validates the document version number, saves the change to the Redis cache, and publishes a synchronization event to the Apache Kafka cluster on the `document-sync` topic.
  4. Active backend instances consume the event from Kafka and broadcast the update to all other users currently viewing the document, keeping their client screens synchronized.

10. User Manual & Operations

10.1 Accessing the Application Platform

The collaborative document editor is accessible via a web browser. Follow these steps to set up a workspace and start editing documents:

Step 1: Sign up and Establish User Profile

Open a web browser and navigate to the application dashboard: https://blockverse.rajjaiswal.online. If you do not have an active profile, click the "Sign Up" button on the login screen. Fill in your name, email, and password to register your account.

Step 2: Workspace Creation and Member Team Organization

Once authenticated, you will be redirected to the main dashboard. To set up a collaborative environment, click the workspace dropdown on the left navigation panel and select "Create Workspace". Define your workspace scope:

  • Personal Workspace: Designed for private document editing and notes.
  • Team Workspace: Designed for multi-user collaboration, allowing you to invite team members by email and assign roles (OWNER, MEMBER).

Step 3: Creating and Editing Collaborative Documents

Inside a workspace, click the "New Document" button to create an editable canvas. The document interface supports standard Notion-like shortcuts:

  • Type `/` to open the quick block selection menu and choose block types like Headings, Todo lists, or Code blocks.
  • Drag and drop blocks to reorder them on the canvas.
  • Share documents with colleagues by clicking "Share" and generating a public, authenticated share link with a configurable expiration date.

11. Production Code Snippets

The following snippets showcase the implementation of the core WebSocket and collaborative editing features in BlockVerse.

11.1 Backend Spring Boot WebSocket Config

WebSocketConfig.java Java 
@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                .setAllowedOrigins("*")
                .withSockJS();
    }

    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.enableSimpleBroker("/topic");
        registry.setApplicationDestinationPrefixes("/app");
    }
}

11.2 Frontend Next.js Collaborative Socket Hook

useDocumentSocket.ts TypeScript 
import { Client } from '@stomp/stompjs';
import { useEffect, useRef } from 'react';

export const useDocumentSocket = (documentId: string, onUpdate: (payload: any) => void) => {
    const stompClientRef = useRef<Client | null>(null);

    useEffect(() => {
        const client = new Client({
            brokerURL: `wss://blockverse.rajjaiswal.online/ws`,
            onConnect: () => {
                client.subscribe(`/topic/document/${documentId}`, (msg) => {
                    onUpdate(JSON.parse(msg.body));
                });
            }
        });
        client.activate();
        stompClientRef.current = client;

        return () => {
            client.deactivate();
        };
    }, [documentId]);
};

12. Future Scope & Conclusion

12.1 Potential Enhancements

While the current implementation of BlockVerse provides a robust, low-latency collaborative document editor, we plan to implement several enhancements in future iterations:

  • Offline-First Sync Engine (CRDTs): Integrate Yjs or Automerge peer-to-peer Conflict-Free Replicated Data Types (CRDTs) to allow seamless offline work with automated merge conflict resolution on reconnect.
  • Richer Block Media Types: Add interactive canvas components, custom database views (Kanban, tables), and media embeds (YouTube, Figma, GitHub Gists).
  • AI Assistant Block: Integrate LLM models to provide in-line writing assistance, grammar correction, summaries, and code block formatting.

12.2 Project Conclusion

BlockVerse demonstrates that a highly performant, real-time collaborative document editor can be built using modern open-source technologies. By combining Java 21's lightweight virtual threads with Next.js Server Components, Redis caching, and Kafka message distribution, the platform achieves sub-100ms sync speeds.

Deploying the entire infrastructure on AWS via Terraform and orchestrating it through a k3s Kubernetes cluster provides a reliable, secure alternative to proprietary SaaS platforms. This project establishes a foundation for organizations requiring direct database control and data privacy.
